# AGENTS.md

Minimum context for any coding agent working in this repo.

## What this is

Adobe Commerce checkout starter kit: a set of independent Adobe Commerce App Management apps that
showcase how to build custom checkout experiences on Adobe Commerce Extensibility + App Builder.
See `README.md` for the per-app business context and links to the official docs.

## Repo structure

npm workspaces monorepo, one root `package.json` and one root `package-lock.json` for everything.

- `apps/*` — one standalone app per checkout domain (`shipping-method`, `payment-method`,
  `tax-integration`, `totals-collector`). Each has its own `package.json`, `app.config.yaml`,
  `install.yaml`, `app.commerce.config.ts`, `src/`, and `test/`.
- `scripts/` — repo-tooling workspace (e.g. CI helper scripts), its own `package.json`/`test/`.
- `.github/workflows/` — CI. `ci.yml` runs root lint+test, then builds/deploys only the apps that
  changed via `apps-pipeline.yml`. See `.github/MAINTAINERS.md` for how the pipeline works and how
  to onboard a new app.

## History

This repo used to be a single monolithic checkout app. It was split into the domain-focused
`apps/*` apps (`shipping-method`, `payment-method`, `tax-integration`, `totals-collector`), each
independently installable/deployable, and the monolithic app was later removed once the split was
complete. Expect no code in the monolithic shape going forward — treat `apps/*` as the only place
checkout functionality lives.

## Apps must stay standalone

Each app under `apps/*` must be independently installable, buildable, and deployable on its own —
that's the whole point of the App Management model (see the README's "App submission guidelines").
Concretely:

- No cross-app imports or shared runtime code between `apps/*` directories. Common needs are
  covered by shared published packages (see below), not by reaching into a sibling app.
- Every app carries its own complete config (`app.config.yaml`, `install.yaml`,
  `app.commerce.config.ts`) and must build/deploy correctly in isolation — assume CI (or a
  reviewer) will run `aio app build`/`aio app deploy` from inside just that one app's directory.
- `src/<extension-point>/.generated/**` is generated by `aio-commerce-lib-app` (the app's
  `postinstall` script runs `aio-commerce-lib-app hooks postinstall`). Don't hand-edit generated
  files — change the source config (e.g. `app.commerce.config.ts`) and regenerate instead.

## Always use `@adobe/aio-commerce-sdk`

Every app depends on `@adobe/aio-commerce-sdk` and `@adobe/aio-commerce-lib-app`. Use them for
Commerce API access, extensibility config (`defineConfig` from
`@adobe/aio-commerce-lib-app/config`), and webhook/action wiring rather than hand-rolling
HTTP calls or config plumbing that these packages already provide.

## Documentation strategy

Keep in-repo docs to the minimum needed to build, test, and deploy. Full install/usage/business
docs are published at
[developer.adobe.com/commerce/extensibility/starter-kit/checkout](https://developer.adobe.com/commerce/extensibility/starter-kit/checkout/),
backed by the [`AdobeDocs/commerce-extensibility`](https://github.com/AdobeDocs/commerce-extensibility/)
repo — that's where user-facing documentation changes belong, not here. Don't grow this repo's
README/app READMEs into a parallel copy of those docs; link out instead.

## Commands

Run from the repo root unless noted — `npm ci` at root installs every workspace's dependencies.

- `npm test` — runs `vitest` across every workspace (all apps + `scripts`).
- `npm run code:check` — runs Biome (`biome check .`) across every workspace.
- `npm run code:fix` — same, with `--write`; also runs automatically via `lint-staged`/husky on
  commit.
- Scope any of the above to one workspace with `-w`, e.g. `npm test -w apps/shipping-method`, or
  `cd apps/shipping-method && npm test`.
- Node version is pinned via `.nvmrc` (see `engines` in each `package.json`).

## Conventions

- Linting: Biome, extending `ultracite/biome/core` (see each app's `biome.jsonc`).
- Tests: Vitest, files under each workspace's `test/**/*.test.js`, coverage via `@vitest/coverage-v8`.
- Module system: `type: module` (ESM) everywhere.
